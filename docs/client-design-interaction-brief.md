# 上海板块写作工作台设计准则与交互方案

版本：2026-04-24  
用途：甲方沟通 / 产品设计说明  
项目定位：本地优先的单人写作工作台

---

## 1. 设计目标

上海板块写作工作台不是传统后台系统，也不是展示型官网。它的核心任务是帮助作者完成一篇上海板块分析文章的完整生产流程：

1. 从选题判断开始，确认这篇文章是否值得写。
2. 收集和整理资料，形成可引用的证据。
3. 把资料转化为板块模型、段落提纲和初稿。
4. 人工改写，并通过质量检查发现硬伤。
5. 整理发布版本，导出 Markdown。

因此，设计目标不是单纯追求装饰感，而是让作者在长时间阅读、判断、修改时保持清楚、稳定、低负担。

界面需要同时满足三件事：

- **清晰**：用户随时知道当前在哪个项目、哪个阶段、下一步该做什么。
- **克制**：资料、判断和正文是主角，界面不抢内容注意力。
- **专业**：整体接近 macOS 生产力工具的稳定感，同时在关键入口和摘要区域保留 Apple 官网式的呼吸感。

---

## 2. 设计方向

本项目采用“Apple-inspired mixed system”，不是直接复制 Apple 官网，也不是照搬原生 App 控件。

### 2.1 高频工作区：macOS 工具感

主工作台、项目列表、资料索引、正文编辑、右侧状态面板，参考 macOS Notes / Finder / Xcode Inspector 的设计思路：

- 左侧是项目索引。
- 中间是当前主任务。
- 右侧是状态与下一步动作。
- 控件安静、清楚，不用强装饰。

### 2.2 低频入口与摘要区：Apple 官网呼吸感

选题共创、新建项目、空状态、项目摘要、证据摘要等区域，允许使用更大的标题、更宽的留白和更集中的主操作，让页面有更好的第一印象和节奏感。

### 2.3 不做的方向

- 不做营销官网式首屏。
- 不把所有内容都做成独立卡片。
- 不用大面积渐变、强阴影、玻璃拟态作为装饰。
- 不把复杂工作流藏进漂亮但不清楚的交互动效里。

## 3. 页面信息架构

桌面端采用稳定三栏结构：

```text
┌──────────────┬──────────────────────────────┬──────────────────┐
│ 项目列表      │ 主工作区                       │ 项目状态面板      │
│              │                              │                  │
│ 搜索 / 最近项目│ 判断 / 资料 / 写作 / 发布       │ 进度 / 风险 / 下一步│
│ 新建 / 共创   │ 当前阶段的主要内容               │ 检查 / 任务 / 导出  │
└──────────────┴──────────────────────────────┴──────────────────┘
```

### 3.1 左侧项目列表

作用：帮助用户快速切换项目，而不是承载复杂编辑。

包含：

- 项目搜索。
- 最近项目。
- 历史项目。
- 测试项目显隐。
- 新建项目入口。
- 选题共创入口。

设计规则：

- 项目标题允许截断，保证列表密度。
- 当前项目用轻蓝色高亮。
- 不在左侧放长内容和复杂表单。

### 3.2 中间主工作区

作用：承载当前最重要的工作对象。

四个主阶段：

| 阶段 | 用户任务 | 主要内容 |
| --- | --- | --- |
| 判断 | 判断文章是否值得写 | 选题评分、ThinkCard、StyleCore、VitalityCheck |
| 资料 | 整理证据 | 研究清单、资料录入、资料索引、证据摘要 |
| 写作 | 生产和修改正文 | 板块建模、段落提纲、双稿编辑、发布整理 |
| 发布 | 输出最终版本 | 发布包、质量检查、Markdown 导出 |

设计规则：

- 同一屏只突出一个主任务。
- 长文阅读和长文编辑不并排放，避免用户左右来回跳。
- 可编辑内容优先采用稳定的单列工作区。
- 资料卡、段落、检查项作为对象卡展示，但不做卡片套卡片。

### 3.3 右侧项目状态面板

作用：像 Inspector 一样给出当前项目状态和下一步建议。

包含：

- 当前流程进度。
- 下一步推荐动作。
- 质量检查状态。
- 风险提示。
- 任务状态摘要。

设计规则：

- 右侧面板固定存在，但视觉权重低于主工作区。
- 不放大段正文，不做长表单。
- 主按钮不铺满整块面板，避免过度抢眼。
- 状态色只用于提醒，不主导页面视觉。

## 4. 核心交互流程

### 4.1 新建项目

入口：左侧“新建”按钮。

交互方式：

- 使用一级弹窗。
- 用户输入主题、文章类型、目标读者等基础信息。
- 弹窗内只保留一个主操作。
- 成功后进入项目工作台。

设计原因：

新建项目是短任务，不需要离开当前工作台；用弹窗可以减少上下文切换。

### 4.2 选题共创

入口：左侧“共创”按钮。

交互方式：

- 使用宽弹窗，展示输入区和候选角度。
- 候选角度以卡片方式排列。
- “选中角度”是小按钮，不作为整张卡片的强主按钮。
- 已选角度有轻量高亮。

设计原因：

共创是创造性入口，需要比普通表单更有呼吸感，但仍然要保持工作台效率。

### 4.3 阶段切换

入口：主工作区顶部的四个阶段 Tab。

交互方式：

- Tab 只改变当前视图，不触发生成或保存。
- 真正会产生数据变化的操作必须使用按钮，例如“生成”“保存”“重新检查”“导出”。

设计原因：

导航和动作分离，避免用户误以为切换页面会自动执行任务。

### 4.4 资料索引

入口：资料阶段中的“资料索引”。

交互方式：

- 顶部展示证据摘要。
- 左侧或上方提示待补证据。
- 资料卡列表采用局部滚动，避免整个页面被资料卡无限拉长。
- 每张资料卡展示标题、状态、可信度、摘要、引用片段、来源链接。

设计原因：

资料索引是“证据桌面”，用户要快速判断哪些资料可用、哪些资料还没进入正文链路。

### 4.5 双稿编辑

入口：写作阶段中的“双稿编辑”。

交互方式：

- 采用单列编辑工作区。
- 人工改写版作为当前主对象。
- 原始生成稿、参考信息、质量提示作为辅助信息放在下方或旁边的低权重区域。
- 不使用左右双栏长文本编辑。

设计原因：

双栏长文编辑会造成用户频繁左右跳读，页面视觉重心不稳定。当前设计优先保证一个明确的写作焦点。

### 4.6 质量检查

入口：右侧项目状态面板或判断 / 发布相关页面。

交互方式：

- 先展示总体状态。
- 再展示必须修、建议修、可选润色。
- 检查结果用小面积状态色和清晰文字解释。
- 不用强红、强黄铺满页面。

设计原因：

质量检查是帮助修稿，不是制造焦虑。视觉上要提醒问题，但不能压过正文。

### 4.7 发布整理

入口：写作阶段的“发布整理”或发布阶段。

交互方式：

- 展示最终发布包。
- 保留导出 Markdown 的明确动作。
- 风险项和未完成项放在导出前提示。

设计原因：

发布阶段要减少干扰，让用户确认结果、处理风险、完成导出。

## 5. 视觉规范

### 5.1 色彩

基础色：

| 类型 | 用途 | 视觉方向 |
| --- | --- | --- |
| 页面背景 | 整体底色 | Apple-like 极浅灰 |
| 主工作区 | 当前内容承载 | 白色大面 |
| 次级区块 | 分组、摘要、工具区 | 浅灰或白色 |
| 主操作色 | 当前选中、链接、主按钮 | Apple blue |
| 状态色 | 成功、风险、失败、过期 | 小面积使用 |

规则：

- 同一屏强状态色不超过两种。
- 蓝色主要用于当前项和主操作。
- 红色只用于失败、删除、阻塞。
- 黄色只用于风险和待处理。
- 绿色只用于成功和通过。
- 不用颜色作为唯一信息，必须配合文字说明。

### 5.2 圆角

| 元素 | 建议圆角 |
| --- | ---: |
| 一级弹窗 | 20px |
| 主舞台 / 大摘要区 | 18px |
| 工具面板 / Inspector | 14px |
| 资料卡 / 项目卡 / 候选卡 | 14px |
| 按钮 / 输入框 | 10px |
| Badge / Chip | 7px |

规则：

- 高频工具控件不过度圆润，避免玩具感。
- 大区块可以更柔和，但不超过 20px。
- 同一类元素保持同一圆角。

### 5.3 字号与字重

| 文本角色 | 字号范围 | 字重 | 用途 |
| --- | ---: | ---: | --- |
| 大标题 | 28-46px | 700 | 共创、阶段标题、重点摘要 |
| 区块标题 | 20-24px | 650-700 | 主工作区区块 |
| 卡片标题 | 16-18px | 650-700 | 资料卡、项目卡 |
| 正文说明 | 14-16px | 400-500 | 普通内容 |
| 元信息 | 12-13px | 500-600 | 标签、状态、时间 |
| 小标签 | 11-12px | 600-700 | Inspector label、badge |

规则：

- 中文正文行宽控制在舒适阅读范围内。
- 不使用很细的字重。
- 不使用负字距。
- 通过字号、字重、颜色建立层级，不靠花色装饰。

### 5.4 阴影与材质

规则：

- 普通卡片和面板不用重阴影。
- 页面层级主要靠留白、灰度、边框、字号建立。
- 只有弹窗、toast、popover 允许轻阴影。
- 默认页面区域不使用渐变。

## 6. 弹窗与浮层规范

### 6.1 一级弹窗

适用场景：

- 新建项目。
- 选题共创。
- 批量导入。
- 发布前确认。

规范：

- 宽度：简单任务 560-720px，复杂任务 960-1040px。
- 圆角：20px。
- 背景：页面轻度遮罩。
- 标题：清楚说明当前任务。
- 主操作：最多一个。
- 关闭按钮：右上角。

禁止：

- 一级弹窗里再开一级弹窗。
- 在弹窗中堆多个同级主按钮。
- 用弹窗承载长期主流程。

### 6.2 二级浮层

适用场景：

- 小筛选。
- 更多操作。
- 简短设置。
- 说明提示。

规范：

- 宽度：280-420px。
- 同时只出现一个。
- 点击外部关闭。
- 内容必须短，不承载复杂编辑。

### 6.3 Toast / 任务提示

适用场景：

- 任务开始。
- 任务完成。
- 任务失败。
- 后台队列状态变化。

规范：

- 出现在不遮挡主工作区的位置。
- 文案短，说明结果和下一步。
- 失败状态必须提供可恢复路径，例如重试或查看任务。

## 7. 组件规范

### 7.1 Surface

用途：页面大区、阶段入口、项目摘要。

规则：

- 大留白。
- 不强调边框。
- 不堆叠多个 surface。

### 7.2 Panel

用途：工具区、右侧状态面板、资料分组。

规则：

- 细边框。
- 轻背景。
- 信息密度适中。

### 7.3 Card

用途：独立对象，例如项目、资料卡、段落、候选角度。

规则：

- 卡片必须代表一个可识别对象。
- 不把普通说明文字都做成卡片。
- 不做卡片套卡片。

### 7.4 Callout

用途：风险、失败、成功、过期提醒。

规则：

- 必须说明原因。
- 必须说明下一步。
- 状态色小面积使用。

### 7.5 Button

规则：

- 每个区域最多一个主按钮。
- 主按钮用于真实动作，例如生成、保存、导出。
- 次按钮用于辅助动作，例如重新生成、查看详情。
- 导航按钮不能看起来像主动作按钮。

## 8. 动效与反馈

动效原则：

- 只服务理解，不制造炫技。
- 切换、出现、hover 可以有 150-200ms 的轻微过渡。
- hover 不做明显位移，避免页面跳动。
- 尊重系统的减少动态效果设置。

反馈原则：

- 后台任务必须有进行中、成功、失败状态。
- 失败必须告诉用户为什么失败，以及可以怎么恢复。
- 保存、生成、导出等动作需要明确结果反馈。

---

## 9. 当前已落地的设计策略

当前版本已经按上述方向完成了以下优化：

- 桌面端保持稳定三栏结构。
- 主工作区成为视觉中心，右侧项目状态面板降低视觉权重。
- 双稿编辑改为单列工作流，减少左右跳读。
- 资料索引增加局部滚动，避免资料卡把整页撑得过长。
- 写作页折叠项头部高度收敛，避免长标题破坏节奏。
- 选题共创弹窗保留更大的呼吸感，同时收敛操作按钮。
- 明显英文界面标签已改为中文表达。
- 普通页面减少渐变和重阴影，采用更克制的 Apple-like surface 系统。

## 10. 验收标准

甲方验收时可以按以下标准判断：

### 10.1 页面结构

- 用户一眼能看出左侧是项目、中间是工作区、右侧是状态。
- 当前阶段和当前子页面明确。
- 页面没有不必要的横向滚动。

### 10.2 视觉一致性

- 同一类按钮、卡片、标签的样式一致。
- 同一屏没有过多强色块。
- 主内容比状态提示更突出。
- 页面有留白，不像传统后台表格堆叠。

### 10.3 交互清晰度

- Tab 只负责切换页面。
- 按钮才触发生成、保存、导出等动作。
- 后台任务有明确反馈。
- 错误状态有可恢复路径。

### 10.4 写作效率

- 长文编辑不左右跳。
- 资料卡可以快速浏览。
- 检查结果能定位问题。
- 发布前能确认风险并导出。

---

## 11. 后续优化方向

后续可继续优化的方向：

1. 为更多状态建立统一的空状态模板。
2. 把所有 badge / chip / status-chip 完全合并为一套组件。
3. 为关键页面建立固定截图验收标准。
4. 增加项目级设计变更记录，避免后续样式再次发散。
5. 根据真实使用反馈，继续压缩右侧状态面板中的低价值信息。

---

## 12. 一句话总结

这个工作台的设计原则是：用 Apple-like 的清晰层级、克制材质和充足留白，服务一个专业写作者从选题、证据、正文到发布的完整工作流；它不是为了展示而设计，而是为了让复杂写作流程更稳定、更可判断、更容易持续推进。
