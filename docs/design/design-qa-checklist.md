# Design QA Checklist

本清单用于每次 UI 改动后的人工或 Codex 验收。

## 1. 结构

- 左侧是否仍是项目索引。
- 中间是否仍是当前主任务。
- 右侧 Inspector 是否只展示状态、风险、下一步。
- 用户是否一眼能看到当前项目、当前阶段、下一步。
- 页面是否无横向滚动。
- 1440px 和 1280px 下是否仍保留桌面 safe area，不贴边、不拥挤。
- 长文阅读和双稿编辑是否在 `--longform-max-width` 内保持单列焦点。
- 表单是否在 `--form-max-width` 内，不横向拉长。

## 2. 视觉

- 背景是否为极浅灰，主内容是否为白色大面。
- 同类按钮、chip、card、panel 是否一致。
- 是否出现了新的重阴影、大面积渐变或玻璃拟态。
- 是否出现卡片套卡片。
- 同屏强状态色是否超过两种。
- 主内容是否比状态提示更突出。
- 标题、正文、说明、meta 是否有清楚层级。
- 字号、行高、字重是否来自 typography token。
- 是否没有新增随机 `padding / gap / border-radius / font-size`。
- 高对比模式下边框、状态、focus 是否更清楚。

## 3. 交互

- Tab 是否只切换视图。
- 生成、保存、导出是否必须由按钮触发。
- 每个区域是否最多一个主按钮。
- 主按钮是否使用 `lg` 强度，工具按钮是否使用 `sm/md`。
- hover 是否不造成明显位移。
- 折叠展开是否自然且不撑乱布局。
- 后台任务是否有开始、进行中、成功、失败反馈。
- 失败是否提供原因和恢复路径。

## 4. 写作效率

- 双稿编辑是否保持单列焦点。
- 原始稿、参考提纲、质量提示是否低于人工稿权重。
- 资料索引是否能快速浏览资料卡。
- 资料卡是否有来源、可信度、摘要、引用片段、状态。
- 质量检查是否先总览，再展示必须修 / 建议修 / 可选润色。

## 5. 弹窗

- 一级弹窗是否只承载短任务。
- 弹窗中是否只有一个主操作。
- 弹窗是否没有再打开另一个一级弹窗。
- 标题和说明是否清楚。
- 关闭按钮是否可见且有 `aria-label`。
- Sheet 是否使用 `kind="sheet"` 和明确 `size`。
- Alert 是否只讲后果和确认，不承载长期表单。
- Popover 是否只承载小筛选/说明，且 Escape 和外部点击可关闭。

## 6. 可访问性

- 按钮是否有明确文本或 `aria-label`。
- Tab 是否有 `role=tablist` / `role=tab` / `aria-selected`。
- Modal 是否有 `aria-labelledby` / `aria-describedby`。
- Toast 是否有 `aria-live`。
- summary / button / tab 是否有可见 focus。
- 状态是否不只靠颜色表达。
- `prefers-reduced-motion` 是否生效。
- `prefers-contrast: more` 是否生效。
- Modal 关闭后焦点是否回到触发按钮。
- 只用键盘是否能走完主阶段和子阶段切换。

## 7. HIG 覆盖

- Layout：主内容是否按阅读顺序从左上开始，控件是否靠近相关内容。
- Color：同一种颜色是否没有同时表达交互和状态。
- Typography：正文是否可读，长文本是否没有过宽或过密。
- Text fields：每个字段是否有可见 label，placeholder 是否只作为例子。
- Lists：项目、资料、任务列表是否有稳定 title / meta / body / actions。
- Search：搜索框是否靠近它影响的列表。
- Materials：内容层、控件层、浮层是否区分清楚，没有玻璃拟态污染正文。

## 8. 桌面截图目标

每轮视觉改动至少检查：

- 1440px：三栏主工作台。
- 1280px：三栏是否仍不挤压。
- 1440px：选题共创弹窗。
- 1440px：资料索引。
- 1440px：双稿编辑。
- 1440px：发布整理。

移动端不是当前主要验收范围，但 narrow fallback 不能出现横向溢出。
